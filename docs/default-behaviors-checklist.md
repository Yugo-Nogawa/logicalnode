# 当たり前実装チェックリスト（デスクトップGUI / CLI）

> 「言われなくても入っているべき」暗黙のUX規約・デファクトスタンダードの網羅リスト。
> vibe codingで抜けがちな“当たり前”を、指示・レビュー・受け入れ確認のチェックリストとして使う。

## 使い方

- 新機能・新ツールを作る前に、関連セクションを **要件として** 読み込ませる。
- Claude Code への指示例：
  > 「この機能を実装するとき、`default-behaviors-checklist.md` の該当セクションを満たすこと。満たせない項目は理由を添えて明示して。」
- 完成後、`- [ ]` を実際に手を動かして確認しながら `- [x]` に変える（テストが通る＝満たす、ではない）。
- 「対象外」は消さずに `- [~] 対象外：理由` として残す。抜けと意図的除外を区別するため。

凡例：🖥 デスクトップGUI固有 / ⌨ CLI固有 / 🌐 両方に効く横断基礎

## 0. 最優先：データを失わせない（🌐）

ユーザーが最も許さないのは「作業が消える」こと。ここだけは妥協しない。

- [x] 未保存の変更がある状態で **閉じる/終了/リロード/別ファイルを開く** 時に警告する（元の例）（※リロード role は未ガード）
- [x] 警告ダイアログは「保存 / 保存しない / キャンセル」の3択（キャンセル＝操作中止できる）
- [x] タイトルバー等に **未保存インジケータ**（例：`● file.txt` や `file.txt — 変更あり`）を出す
- [ ] クラッシュ・強制終了に備えた **自動保存 / 下書き復元**（一定間隔 or 変更ごと）
- [x] 上書き保存の前に **一時ファイルへ書いてから rename**（書き込み途中のクラッシュで原本を壊さない）
- [x] 破壊的操作（削除・上書き・一括変更）は **確認 or undo可能** のどちらかを必ず用意（削除は複数段undoで回復可）
- [x] 既存ファイルを上書きする時は確認する（⌨ なら `--force` を要求 / GUI なら上書き確認ダイアログ）→ 詳細は §17
- [~] 対象外：長時間バッチ処理が無い（原本は原子的書き込みで保護済み）
- [x] エクスポート/インポートで元データを変更しない（読み取り専用として扱う）

## 17. 出力の上書き防止（アウトプット衝突対策）（🌐）

- [x] ⑤ 原子的書き込み：一時ファイル→`rename`。上書きが意図的でも、途中失敗で原本を壊さない

**堅牢化（レース・特殊ケース）**
